# Design: 반도체 답변에 관련 안전보건 내용 자동 첨부

> **Feature**: 반도체답변-안전보건-자동첨부
> **Date**: 2026-03-13
> **Status**: Design
> **Plan**: [반도체답변-안전보건-자동첨부.plan.md](../../01-plan/features/반도체답변-안전보건-자동첨부.plan.md)

---

## 1. 설계 개요

반도체 도메인(`semiconductor-v2`) 답변 생성 시, `kosha` 네임스페이스에서 관련 안전보건 문서를 자동으로 크로스 검색하여 LLM 프롬프트에 보조 컨텍스트로 주입한다.

**기존 패턴 활용**: `laborlaw` 네임스페이스의 Law API 크로스 소스 패턴(`run_rag_pipeline()` → `law_references` → `build_llm_prompts()` 주입)을 반도체→kosha 크로스 검색에 적용.

## 2. 변경 파일 및 상세 설계

### 2.1 `services/rag_pipeline.py`

#### 2.1.1 새 함수: `_search_safety_context()`

**위치**: `run_rag_pipeline()` 앞, `_run_legal_analysis_pass()` 다음 (약 line 202 부근)

```python
# -- Configuration --
SAFETY_CROSS_SEARCH_TOP_K = 5
SAFETY_CROSS_SEARCH_MIN_SCORE = 0.3
SAFETY_CROSS_SEARCH_MAX_RESULTS = 3
SAFETY_CROSS_SEARCH_MAX_CHARS = 1500
SAFETY_CROSS_SEARCH_NAMESPACE = 'kosha'


def _search_safety_context(query: str) -> str:
    """Cross-search kosha namespace for safety/health content related to semiconductor query.

    Args:
        query: Original user query (or search_query after mention parsing).

    Returns:
        Formatted safety references string for prompt injection.
        Empty string if no relevant results found.
    """
```

**동작 흐름**:
1. `get_agent().search(query=query, top_k=SAFETY_CROSS_SEARCH_TOP_K, namespace=SAFETY_CROSS_SEARCH_NAMESPACE)` 호출
2. 결과를 `score >= SAFETY_CROSS_SEARCH_MIN_SCORE`로 필터링
3. 상위 `SAFETY_CROSS_SEARCH_MAX_RESULTS`개만 선택
4. 결과가 0개이면 빈 문자열 반환 (섹션 자체 생략)
5. 각 결과를 `[S{i}] (출처: {source_file})\n{content}` 형식으로 포맷
6. 전체 텍스트가 `SAFETY_CROSS_SEARCH_MAX_CHARS`자를 넘지 않도록 자르기
7. 포맷된 문자열 반환

**에러 처리**: 전체를 `try/except`로 감싸서, 실패 시 빈 문자열 반환 + `logging.warning()`. 기존 답변에 영향 없음 (graceful degradation).

#### 2.1.2 `run_rag_pipeline()` 수정

**위치**: Phase 7 완료 후, `result` dict 구성 후, `if namespace == 'laborlaw':` 블록 앞 (약 line 615 부근)

```python
    # ========================================
    # Phase 7.5: Safety Cross-Search (semiconductor only)
    # ========================================
    if namespace in ('', 'semiconductor-v2'):
        try:
            safety_refs = _search_safety_context(search_query)
            if safety_refs:
                result['safety_references'] = safety_refs
                logging.info("[SafetyCross] Found safety context (%d chars)", len(safety_refs))
        except Exception as e:
            logging.warning("[SafetyCross] Failed: %s", e)
```

**설계 결정**:
- `namespace == ''`도 포함: 기본(semiconductor) 네임스페이스가 빈 문자열(`''`)인 경우도 있으므로 (NAMESPACE_DOMAIN_MAP에서 `'semiconductor-v2' → 'semiconductor'`이지만 실제 라우팅은 namespace 값으로 판단)
- `search_query` 사용 (mention 파싱 후 클린 쿼리): 원본 query 대신 파싱된 쿼리로 검색하여 정확도 향상

#### 2.1.3 `build_llm_prompts()` 수정

**시그니처 변경**:
```python
def build_llm_prompts(query, sources, context, namespace, calc_result=None,
                      law_references=None, labor_classification=None,
                      legal_analysis=None, safety_references=None):
```

**주입 위치**: `legal_analysis` 주입 블록 아래, `if namespace == 'laborlaw':` 블록 위 (약 line 710~720 사이)

```python
    # Inject safety/health cross-search results for semiconductor domain
    if safety_references:
        user_prompt += f"""

## 관련 안전보건 참고자료
{safety_references}

위 안전보건 자료를 참고하여, 답변 말미에 "⚠️ 관련 안전보건 정보" 섹션을 추가하세요.
해당 공정/물질의 유해물질, 보호구, 안전수칙, 응급조치 등을 간결하게 안내하세요.
관련성이 낮으면 이 섹션을 생략하세요."""
```

**설계 결정**:
- `safety_references` 파라미터가 truthy일 때만 주입 → 결과 없으면 자동 스킵
- LLM에게 "관련성 낮으면 생략" 지시 → 이중 안전장치 (score 필터 + LLM 판단)
- 인용 번호를 `[S1]`, `[S2]`로 사용 → 기존 `[1]`, `[2]` 소스 인용과 충돌 방지

#### 2.1.4 `build_llm_messages()` 수정

**시그니처에 `safety_references=None` 추가**하고, 내부 `build_llm_prompts()` 호출에 전달:

```python
def build_llm_messages(query, sources, context, namespace, calc_result=None,
```

## 3. 데이터 흐름

```
사용자 쿼리: "CVD 공정 원리는?"
    │
    ▼
Phase 1-7: 기존 RAG 파이프라인 (semiconductor-v2 검색)
    │  → context, sources 확보
    │
    ▼
Phase 7.5: _search_safety_context("CVD 공정 원리는?")
    │  → kosha 네임스페이스 벡터 검색 (top_k=5)
    │  → score >= 0.3 필터링
    │  → 상위 3개 결과 포맷
    │  → safety_references 문자열 반환
    │
    ▼
result['safety_references'] = safety_refs
    │
    ▼
api/v1/search.py: build_llm_messages(..., safety_references=safety_refs)
    │
    ▼
build_llm_prompts(): user_prompt에 "## 관련 안전보건 참고자료" 섹션 주입
    │
    ▼
Phase 8: LLM 답변 생성
    │  → 기술 답변 + "⚠️ 관련 안전보건 정보" 섹션
    │
    ▼
사용자에게 통합 답변 표시
```

## 5. 설계 결정 근거

| 결정 | 이유 |
|------|------|
| `[S1]`, `[S2]` 인용 형식 | 기존 `[1]`, `[2]` 소스 인용과 충돌 방지. LLM이 두 인용 체계를 구분 가능. |
| `search_query` 사용 (원본 query 대신) | @mention 파싱 후 클린 쿼리로 검색하여 더 정확한 안전 관련 결과 반환 |
| `namespace in ('', 'semiconductor-v2')` 조건 | 반도체 도메인의 namespace가 빈 문자열 또는 `semiconductor-v2`일 수 있음 (기존 코드에서 `DEFAULT_NAMESPACE = ''`) |
| 최소 score 0.3 | 기존 `MIN_RELEVANCE_SCORE` (0.2)보다 높게 설정하여 크로스 도메인 노이즈 방지 |
| 최대 3개 결과 | 토큰 예산(+500~1500) 제한 + 너무 많은 안전 정보는 기술 답변을 희석 |
| 최대 1500자 | LLM 컨텍스트 윈도우 부담 최소화 |
| graceful degradation | `try/except`로 감싸서 크로스 검색 실패 시 기존 기술 답변만 제공 |
| `build_llm_prompts()`에서 주입 | `run_rag_pipeline()`가 아닌 프롬프트 빌더에서 주입하여 관심사 분리 유지 |
| `safety_references` 파라미터 기본값 `None` | 기존 호출 코드 하위 호환 유지 — 안전 참고자료 없으면 아무 영향 없음 |

---

## 6. 비목표 (Not Goals)

- 안전보건 결과를 `sources` 배열에 합치지 않음 → 프론트엔드 소스 카드에 표시할 필요 없음
- `safety_references`를 SSE `metadata` 이벤트에 포함하지 않음 → 프롬프트에만 주입
- 다른 도메인(field-training, msds)에 크로스 검색 적용하지 않음 → 이번 범위 아님
- 병렬 검색 최적화(메인 검색과 kosha 동시 실행) → P3으로 향후 개선
