# Plan: @mention 삭제 & 질문 기반 자동 네임스페이스 라우팅

> **Feature**: 자동-네임스페이스-라우팅
> **Date**: 2026-03-13
> **Status**: Plan

---

## Executive Summary

| 관점 | 설명 |
|------|------|
| **Problem** | 사용자가 질문 시 `@파일명` `@폴더명/` 형태로 문서 범위를 수동 지정해야 하는 UX가 비직관적이고 사용률이 낮음. 또한 사용자가 어떤 도메인 페이지에서 질문해야 할지 미리 알아야 하는 불편. |
| **Solution** | @mention 시스템을 완전 삭제하고, 질문 텍스트를 분석하여 적합한 네임스페이스를 자동으로 결정하는 도메인 라우터를 구현. 각 도메인 페이지에서는 기존처럼 해당 도메인을 기본 검색하되, 질문이 다른 도메인에 해당하면 자동 전환. |
| **Function & UX** | 사용자가 반도체 페이지에서 "최저임금 위반 여부"를 질문하면 → laborlaw 네임스페이스로 자동 라우팅하여 정확한 답변 제공. @mention 학습 비용 제거. |
| **Core Value** | 검색 진입 장벽을 낮추고, "어디서 질문하든 최적의 답변"이라는 SafeFactory의 통합 지식 플랫폼 비전 실현. |

## 1. 배경 및 문제

### 1.1 현재 상태

- **@mention 시스템**: `@파일명`, `@폴더명/`, `@키워드`로 검색 범위 지정 (filters.py `parse_mentions()`)
- **도메인 페이지 분리**: 반도체(/semiconductor), 노동법(/laborlaw), 현장실습(/field-training), 안전보건(/safeguide), MSDS(/msds) 각각 별도 페이지
- **namespace 고정**: 각 도메인 페이지에서 질문하면 해당 namespace로만 검색 (domain.html `DEFAULT_NAMESPACE`)

### 1.2 문제점

| 문제 | 영향 |
|------|------|
| @mention 사용률 극히 낮음 | UI에 설명이 있지만 대부분 사용자가 무시 — 학습 비용 대비 효용 낮음 |
| 도메인 선택 실수 | 반도체 페이지에서 노동법 질문 → 결과 없음 → 사용자가 페이지 이동해야 함 |
| @mention 코드 복잡도 | Phase 3 mention filtering, search_top_k 조정, loadSources API 등 불필요한 복잡도 |
| 통합 검색(all) 품질 낮음 | 모든 네임스페이스를 동시에 검색하면 관련 없는 결과 혼재 |

### 1.3 기존 유사 패턴

`query_router.py`에 이미 질문 유형 분류기(`classify_query_type()`)가 regex 기반으로 존재. 이 패턴을 도메인 분류에 확장 적용.

## 2. 목표

### 2.1 핵심 목표

1. `@mention` 관련 코드 완전 삭제 (parse_mentions, build_source_filter, Phase 3 mention filtering, loadSources API, 프론트엔드 UI 힌트)
2. 질문 텍스트 기반 자동 도메인 분류기 `classify_domain()` 구현
3. 각 도메인 페이지에서 해당 도메인을 기본으로 사용하되, 질문이 명확히 다른 도메인에 해당하면 자동 전환
4. 검색 결과에 "감지된 도메인" 정보를 포함하여 사용자에게 피드백

### 2.2 비목표 (Not Goals)

- 모든 도메인 페이지를 하나로 통합 (기존 도메인 페이지 구조는 유지)
- LLM 기반 질문 분류 (regex/keyword 기반으로 충분, 레이턴시 0ms 유지)
- 멀티 도메인 동시 검색 (단일 최적 도메인에 라우팅)

## 3. 기술 접근 방식

### 3.1 도메인 분류기 (Domain Classifier)

```
사용자 질문: "CVD 공정에서 사용하는 화학물질의 MSDS 정보"
    ↓
classify_domain(query, default_namespace)
    ├── 키워드 매칭: "CVD 공정" → semiconductor (score: 0.5)
    ├── 키워드 매칭: "MSDS" → msds (score: 0.8)
    ├── 키워드 매칭: "화학물질" → kosha (score: 0.3)
    ├── default_namespace 가중치: semiconductor (score: 0.3)
    ↓
최종 결정: msds (highest combined score)
```

**분류 전략**:
- 각 도메인별 키워드 사전 정의 (고유 키워드 = 높은 가중치, 공통 키워드 = 낮은 가중치)
- `default_namespace`(현재 페이지의 도메인)에 보너스 가중치 부여 → 페이지 컨텍스트 존중
- 모든 도메인 score가 낮으면(확신 부족) → default_namespace 유지
- 반환: `(namespace, confidence, detected_domain_label)`

### 3.2 도메인 키워드 사전 (초안)

| 도메인 | 고유 키워드 (가중치 높음) | 공통 키워드 (가중치 낮음) |
|--------|--------------------------|--------------------------|
| semiconductor-v2 | CVD, PVD, 에칭, 리소그래피, 웨이퍼, 패키징, 반도체, NCS, 포토공정, CMP, 이온주입, 스퍼터링, 증착, 식각, 도핑 | 공정, 제조, 장비 |
| laborlaw | 근로기준법, 최저임금, 퇴직금, 4대보험, 연차, 부당해고, 임금, 연봉, 실수령, 세후, 주휴수당, 해고, 노동법, 근로계약 | 법률, 법적, 위반 |
| field-training | 현장실습, 카드북, 실습생, 직업계고 | 안전수칙, 재해, 보호구 |
| kosha | 안전보건공단, KOSHA, 산업재해, 안전관리, 작업환경측정 | 안전, 보건, 위험, 보호구 |
| msds | MSDS, CAS, 물질안전보건자료, GHS, H코드, 화학물질명(벤젠, 톨루엔 등) | 화학물질, 유해성 |

| 삭제 대상 | 파일 | 설명 |

### 3.4 RAG 파이프라인 수정

```
기존 Phase 1-7:
  Phase 1: Query Enhancement (with @mention parsing)
  Phase 2: Search (mention_filters → search_top_k 조정)
  Phase 3: Apply mention filters  ← 삭제
  Phase 4: Hybrid Search
  ...

변경 후:
  Phase 0 [NEW]: Domain Classification
    ├── classify_domain(query, default_namespace)
    ├── namespace 자동 결정
    └── detected_domain 정보 반환
  Phase 1: Query Enhancement (단순화: @mention 파싱 제거)
  Phase 2: Search (mention_filters 관련 코드 삭제)
  Phase 3: ← 기존 Phase 4로 번호 당겨짐 (Hybrid Search)
  ...
```

## 5. 위험 요소

| 위험 | 영향 | 완화 |
|------|------|------|
| 도메인 오분류 | 질문이 엉뚱한 namespace로 라우팅 | default_namespace 보너스 + confidence 낮으면 기본 유지 |
| 다중 도메인 질문 | "CVD 공정 MSDS는?" 반도체+MSDS 혼합 | 가중치 합산으로 최적 단일 도메인 선택 |
| 기존 @mention 사용자 불편 | 파워유저가 @mention 사용 중 | 사용률이 극히 낮아 영향 최소 |
| 도메인 키워드 누락 | 새 키워드가 분류되지 않음 | default_namespace로 fallback + 키워드 사전 쉽게 확장 가능 |
