## 1. 아키텍처

### 1.1 변경 전 데이터 흐름

```
사용자 입력: "@패키지/ CVD 공정 설명해줘"
    ↓
parse_mentions(query)
    → clean_query: "CVD 공정 설명해줘"
    → mention_filters: [{type: 'folder', value: '패키지'}]
    ↓
Phase 1: Query Enhancement (clean_query 기반)
    ↓
Phase 2: Search (mention_filters → search_top_k * TOP_K_MENTION_MULT)
    ↓
Phase 3: Apply mention filters (Python in-memory filtering)
    ↓
Phase 4-7: Hybrid → Rerank → Optimize → Context
```

### 1.2 변경 후 데이터 흐름

```
사용자 입력: "최저임금 위반 여부" (반도체 페이지에서)
    ↓
Phase 0 [NEW]: classify_domain(query, default_namespace='semiconductor-v2')
    → detected_namespace: 'laborlaw'
    → confidence: 0.85
    → domain_label: '노동법'
    ↓
Phase 1: Query Enhancement (query 그대로 — @mention 파싱 불필요)
    ↓
Phase 2: Search (namespace=detected_namespace, top_k 계산 단순화)
    ↓
Phase 3: ← 삭제됨 (기존 Phase 4가 이 자리)
    ↓
Phase 3~6 (기존 4~7): Hybrid → Rerank → Optimize → Context
    ↓
result['detected_namespace'] = 'laborlaw'  ← 프론트엔드 피드백용
```

### 1.3 핵심 컴포넌트

| 컴포넌트 | 위치 | 역할 |
|----------|------|------|
| `classify_domain()` | `services/query_router.py` | 질문 키워드로 최적 namespace 결정 |
| `DOMAIN_KEYWORDS` | `services/query_router.py` | 도메인별 키워드 사전 (가중치 포함) |
| Phase 0 로직 | `services/rag_pipeline.py` | classify_domain 호출 + namespace 오버라이드 |

### 2.2 `classify_domain()` 함수 설계

```python
# 가중치 상수
_HIGH_KEYWORD_WEIGHT = 1.0
_LOW_KEYWORD_WEIGHT = 0.3
_DEFAULT_NS_BONUS = 0.5      # 현재 페이지 도메인 보너스
_CONFIDENCE_THRESHOLD = 0.3   # 이 점수 미만이면 default 유지

def classify_domain(query: str, default_namespace: str = '') -> tuple:
    """질문 텍스트를 분석하여 최적 네임스페이스를 결정.

    Args:
        query: 사용자 질문 텍스트.
        default_namespace: 현재 페이지의 기본 네임스페이스.

    Returns:
        tuple: (namespace, confidence, domain_label)
        - namespace: 최적 Pinecone namespace (str)
        - confidence: 분류 확신도 0.0~1.0 (float)
        - domain_label: 사용자 표시용 도메인 이름 (str)
    """
```

**분류 알고리즘**:

1. 질문을 소문자 변환 (대소문자 무관 매칭)
2. 각 도메인별 키워드 매칭 → 가중치 합산 점수 계산
3. `default_namespace`에 보너스 가중치 추가
4. 최고 점수 도메인 선택
5. 최고 점수 < `_CONFIDENCE_THRESHOLD` → default_namespace 유지
6. `(namespace, confidence, domain_label)` 반환

**domain_label 매핑**:

```python
_NAMESPACE_LABELS = {
    'semiconductor-v2': '반도체',
    'laborlaw': '노동법',
    'field-training': '현장실습',
    'kosha': '안전보건',
    'msds': 'MSDS',
}
```

### 2.4 프론트엔드 응답 확장

`api/v1/search.py` — `api_ask()`, `api_ask_stream()` 응답에 추가:

```python
# api_ask() 응답에 추가
resp_data['detected_namespace'] = pipeline.get('detected_namespace')

# api_ask_stream() metadata event에 추가
meta_data['detected_namespace'] = pipeline.get('detected_namespace')
```

프론트엔드에서는 `detected_namespace`가 현재 페이지의 namespace와 다르면 "💡 이 질문은 {domain_label} 영역에서 답변되었습니다" 같은 피드백을 표시할 수 있음 (프론트엔드 변경은 이번 스코프에서 optional).

## 5. 테스트 시나리오

### 5.1 도메인 분류 정확도

| 질문 | 현재 페이지 | 기대 namespace | 비고 |
|------|------------|----------------|------|
| "CVD와 PVD 차이점" | semiconductor | semiconductor-v2 | 기본 + 고유키워드 |
| "최저임금 위반 여부" | semiconductor | laborlaw | 다른 도메인 감지 |
| "최저임금 위반 여부" | laborlaw | laborlaw | 기본 + 고유키워드 |
| "현장실습 안전수칙" | semiconductor | field-training | 다른 도메인 감지 |
| "벤젠 MSDS 정보" | kosha | msds | MSDS 고유키워드 |
| "작업환경 안전관리" | semiconductor | semiconductor-v2 | low 키워드 → default 유지 |
| "안전보건공단 가이드" | semiconductor | kosha | KOSHA 고유키워드 |
| "안녕하세요" | semiconductor | semiconductor-v2 | 키워드 없음 → default 유지 |

### 5.2 기존 기능 호환

- 각 도메인 페이지(`/semiconductor`, `/laborlaw`, `/field-training`, `/safeguide`, `/msds`)에서 해당 도메인 질문 시 기존과 동일하게 동작
- `namespace=all` 전체 검색은 분류 대상에서 제외 (기존 로직 유지)
- @mention 없는 일반 질문은 기존과 동일한 결과
